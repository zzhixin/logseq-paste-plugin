# Logseq Paste Plugin 方案文档

## 1. 目标

开发一个 Logseq 插件，专门处理从剪贴板粘贴到 Logseq 编辑器的内容，重点解决如下问题：

1. 当剪贴板内容中包含“文本 + base64 图片”时，不直接把 base64 原文粘贴进 block。
2. 自动从 base64 中提取图片并写入当前图谱的附件目录。
3. 在当前 block 中插入图片引用，而不是插入体积巨大的 base64 文本。
4. 尽量保留原始文本内容，让最终粘贴结果更接近用户预期。

## 2. 需求拆解

### 2.1 输入形态

插件需要优先支持以下剪贴板文本场景：

1. 纯文本中包含 `data:image/...;base64,...` 片段。
2. Markdown/HTML 文本中包含 base64 图片，例如：
   - `![alt](data:image/png;base64,...)`
   - `<img src="data:image/png;base64,...">`
3. 一段普通文字前后夹杂一个或多个 base64 图片。

### 2.2 输出形态

处理后的 block 内容应符合 Logseq 使用习惯：

1. 普通文字原样保留。
2. 每个 base64 图片转换成附件文件。
3. 在文本中插入 Logseq 图片引用，例如：
   - `![](../assets/xxx.png)` 或
   - `![](/assets/xxx.png)`

具体引用格式在实现阶段以 Logseq 当前图谱的实际附件路径表现为准。

## 3. 设计原则

### 3.1 最小侵入

只接管粘贴到 Logseq 编辑器内的行为，不影响系统剪贴板本身，也不试图改写其他应用的粘贴。

### 3.2 安全回退

如果解析失败、落盘失败、无法定位编辑器，插件应回退到默认粘贴行为，避免“按下粘贴后什么都没有”。

### 3.3 可逐步增强

第一阶段先支持最核心的文本 base64 场景；后续可以再增强 HTML 粘贴、图片命名策略、用户配置项等能力。

## 4. 技术方案

## 4.1 总体流程

1. 插件启动后注册粘贴监听。
2. 用户在 Logseq 编辑器中执行粘贴。
3. 插件读取本次剪贴板中的文本内容。
4. 判断文本中是否存在 base64 图片片段。
5. 如果不存在：
   - 不接管，继续默认粘贴。
6. 如果存在：
   - 阻止默认粘贴。
   - 解析文本中的 base64 图片。
   - 将每张图片写入附件目录。
   - 生成对应的图片引用 Markdown。
   - 用“普通文本 + 图片引用”的新内容写回当前编辑中的 block。

## 4.2 粘贴接管策略

优先采用 DOM 层监听编辑器 `paste` 事件的方式实现，因为需求本质上是拦截编辑区的用户输入行为。

候选实现：

1. 在 `document` 上注册捕获阶段的 `paste` 事件监听。
2. 判断事件目标是否位于 Logseq 编辑器内部。
3. 从 `ClipboardEvent` 中读取：
   - `text/plain`
   - 必要时补充读取 `text/html`
4. 只有检测到目标内容包含 base64 图片时才 `preventDefault()`。

这样可以避免影响正常文本粘贴，也避免每次粘贴都走自定义流程。

## 4.3 base64 识别策略

第一阶段采用正则 + 分类型解析。

### 识别模式

1. 通用 data URL：
   - `data:image/png;base64,...`
   - `data:image/jpeg;base64,...`
   - `data:image/gif;base64,...`
   - `data:image/webp;base64,...`
2. Markdown 图片：
   - `!\\[.*?\\]\\((data:image\\/[^)]+)\\)`
3. HTML 图片：
   - `<img[^>]+src=["'](data:image\\/[^"']+)["'][^>]*>`

### 解析结果结构

建议内部统一为：

```ts
type ParsedSegment =
  | { type: "text"; content: string }
  | {
      type: "image";
      mimeType: string;
      extension: string;
      base64Data: string;
      originalSource: string;
      alt?: string;
    };
```

这样后续“写文件”和“重组输出文本”会更清晰。

## 4.4 图片文件生成策略

### 文件目录

优先写入当前 graph 的附件目录。目录来源按以下优先级处理：

1. 读取 Logseq 当前图谱配置中的附件目录。
2. 如果拿不到配置，则回退到常见默认目录，例如 `assets/`。

### 文件命名

建议使用时间戳 + 短随机串，避免冲突，例如：

`paste-20260502-150301-ab12cd.png`

### 扩展名映射

根据 MIME 类型决定扩展名：

1. `image/png` -> `.png`
2. `image/jpeg` -> `.jpg`
3. `image/gif` -> `.gif`
4. `image/webp` -> `.webp`
5. 未识别类型回退到 `.bin` 或直接拒绝处理

第一阶段建议只允许常见图片类型，降低不确定性。

## 4.5 文件写入策略

核心步骤：

1. 去掉 data URL 头部，仅保留 base64 数据。
2. 将 base64 转换为二进制。
3. 使用 Logseq 提供的文件写入能力，或在插件可访问范围内写入图谱附件目录。
4. 写入成功后得到相对路径。

需要单独封装：

```ts
async function saveBase64Image(input: {
  mimeType: string;
  base64Data: string;
}): Promise<{ relativePath: string }>
```

## 4.6 block 内容替换策略

目标不是直接操作原始文件，而是更新当前编辑器中的内容。

候选方式：

1. 如果能拿到当前编辑中的 block UUID，则直接通过 Logseq Editor API 更新 block。
2. 如果更适合在 DOM 编辑器中插入文本，则将转换后的文本插入到当前光标位置。

优先级建议：

1. 优先“替换当前粘贴动作的插入内容”。
2. 若 API 限制较大，再退而求其次为“拿到当前 block 后整体重写”。

最终输出示例：

```md
这是原始说明文字

![](../assets/paste-20260502-150301-ab12cd.png)

这是图片后面的补充说明
```

如果同一段中有多张图片，则按原出现顺序生成多个引用。

## 4.7 错误处理

以下情况必须显式处理：

1. 剪贴板没有文本数据。
2. 文本中没有合法 base64 图片。
3. base64 解码失败。
4. 附件目录不存在且无法创建。
5. 文件写入失败。
6. 当前没有可编辑 block。

处理原则：

1. 输出日志到插件控制台。
2. 使用 `logseq.UI.showMsg` 给用户简短提示。
3. 尽量回退到默认粘贴，而不是吞掉内容。

## 5. 模块划分

建议拆成以下模块：

1. `src/main.ts`
   - 插件入口
   - 注册/注销粘贴事件
2. `src/paste-handler.ts`
   - 粘贴事件处理主流程
3. `src/parser.ts`
   - 解析文本中的 base64 图片和普通文本
4. `src/file-service.ts`
   - 附件目录解析
   - base64 图片写入
5. `src/editor-service.ts`
   - 将最终结果插入或写回当前 block
6. `src/types.ts`
   - 公共类型定义

## 6. 分步实施计划

### 第一步：初始化插件骨架

目标：

1. 建立 `package.json`、TypeScript、构建脚本。
2. 建立基础 `src/main.ts`。
3. 能在 Logseq 中加载插件。

### 第二步：实现纯解析能力

目标：

1. 输入一段带 base64 的字符串。
2. 输出结构化的文本段与图片段。
3. 先用单元测试或日志验证解析结果。

### 第三步：实现图片落盘

目标：

1. 将单个 base64 图片保存到附件目录。
2. 返回可引用路径。
3. 验证生成文件是否能被 Logseq 正常显示。

### 第四步：接入粘贴事件

目标：

1. 只在检测到 base64 图片时接管粘贴。
2. 普通文本仍保持默认行为。
3. 发生异常时能优雅回退。

### 第五步：整体验证

验证以下场景：

1. 纯文本粘贴不受影响。
2. 单张 base64 图片可转成附件并引用。
3. 文本 + 单张图片可按顺序插入。
4. 文本 + 多张图片可按顺序插入。
5. 非法 base64 不会破坏正常粘贴。

## 7. 预期风险

### 7.1 Logseq 编辑器 API 差异

不同版本 Logseq 对当前编辑 block、插入文本、附件目录配置的可访问方式可能存在差异。编码阶段需要以当前可用 API 为准，必要时采用 DOM 事件 + 编辑器 API 混合方案。

### 7.2 剪贴板内容格式不统一

某些来源应用复制出来的内容可能是 HTML 优先、纯文本兜底，甚至会把图片同时放在 `ClipboardItem` 和文本里。第一阶段先解决“文本中自带 base64”的主要路径。

### 7.3 大图片性能

极大的 base64 字符串在解析和写盘时会带来卡顿。第一阶段建议先串行处理，后续再考虑体积限制和异步提示。

## 8. 第一阶段边界

本阶段先明确不做：

1. OCR
2. 图片压缩
3. 远程图片下载
4. 非图片类型的 data URL
5. 对历史 block 中已有 base64 的批量迁移

## 9. 下一步建议

按文档的实施计划推进，先完成插件工程骨架，再补解析器和落盘能力，最后接入粘贴拦截。
